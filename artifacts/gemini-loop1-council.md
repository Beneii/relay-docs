# Loop 1 Council: Highest-Impact Work — Gemini Council

1. **Integrated Delivery Logs & Notification History UI:** Build a "Logs" tab in the dashboard that surfaces the last 50 webhook attempts, including request payload, timestamp, HTTP status code, and Expo push delivery result (sent/failed). For vibe-coders building complex agent loops, push notifications are currently a "black box"; without these logs, they cannot debug whether a missing alert is due to their code, a rate limit, or a Relay internal error, leading to immediate churn when "it just doesn't work."

2. **Heartbeat "Silence" Monitoring (Dead-man's Switch):** Implement a configuration option per dashboard to expect a "Heartbeat" webhook at a defined interval (e.g., every 15 minutes). If Relay doesn't receive a ping within that window, it triggers a high-severity "System Offline" notification to the user's phone. This is a killer feature for AI agent builders and trading bot developers who need to know the *instant* their autonomous tool crashes, providing unique product substance that differentiates Relay from generic notification pipes like ntfy or Pushover.

3. **HMAC Signing for Action Button Callbacks:** Add a security layer where Relay signs all POST requests triggered by mobile action buttons using an HMAC-SHA256 signature (generated with the app's `webhook_token` as the secret) in a `x-relay-signature` header. As users move from simple alerts to "Contextual Actions" like `Approve Trade` or `Unlock Door`, they need a robust, production-grade way to verify that the command truly originated from their Relay app and wasn't spoofed, directly addressing the biggest security gap in the current "Mobile Runtime" pivot.
