# Loop 9 Council: Mobile Notification History — Gemini Council

### Highest-Risk Design Decisions
1. **Real-time vs. Polling (The "Buzz-to-Inbox" Gap):** Deciding between Supabase Real-time and simple polling/Refetch-on-Focus is a high-stakes trade-off. While Real-time provides a "magic" feel, it significantly increases battery drain and connection overhead in mobile environments. Given that the primary "buzz" is handled by the OS push service, a hybrid approach—refreshing the list only when the app is focused or a notification is tapped—is lower risk but must handle the "Cold Start" case perfectly to ensure the user sees the alert they just tapped.
2. **Infinite Scroll vs. Pro Tier Gating:** The current implementation in `notifications.tsx` uses a hard limit (`FREE_VISIBLE_LIMIT = 20`) and `slice()`. This is inefficient as it fetches 100 items but only shows 20. Shifting to true server-side pagination with an `InfiniteQuery` is necessary for performance, but we must decide if the "Pro" gate happens at the *query* level (limiting the database results) or the *UI* level (stopping the scroll). Blocking at the query level is more secure and performant.
3. **Optimistic "Read" State:** Mobile users expect immediate feedback. When tapping a notification or "Mark all as read," the UI must update instantly using React Query's `onMutate` optimistic updates. Relying on the Supabase round-trip will make the app feel "laggy" on cellular networks, especially since `read_at` updates are frequent.

### Edge Cases to Watch
*   **The "Vanish" Bug (Push vs. DB):** A notification arrives via Apple/Google Push, the user taps it, the app opens, but the Supabase query hasn't indexed the new row yet (or the network is slow). The app must have a fallback "loading" or "retry" state for the specific `notificationId` passed from the push payload to avoid showing an "Alert not found" error.
*   **Action Button Synchronicity:** If a user triggers an action button directly from the notification tray (foreground or background), the in-app history needs to reflect that state change (e.g., "Action: Approved") without requiring a manual pull-to-refresh.
*   **Deep Link Path Collision:** If a notification contains a `deep_link_url` that opens a specific app view, the app must decide if it *also* marks that notification as read in the background. If it doesn't, the user will see an "unread" badge on the tab for a notification they just acted upon.
*   **Offline Receipt Persistence:** If the user marks a notification as read while offline, the app should queue that change in a local `storageAdapter` (using `expo-secure-store`) and sync it when connectivity returns, rather than losing the "read" state on app close.
